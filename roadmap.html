MedArchive — автоматическая обработка прайсов клиник

Система принимает архив прайс-листов клиник-партнёров в разных форматах, извлекает услуги и цены, приводит их к единому справочнику услуг, проверяет, версионирует и отдаёт через REST API: «кто оказывает услугу и по какой цене».

Стек: Python · FastAPI · PostgreSQL · Celery/Redis · pdfplumber · Tesseract OCR · RapidFuzz + sentence-transformers

1Проблема и цель

У партнёров прайсы в PDF, сканах, Word и Excel — разной структуры, с шапками-«шапками», тарифами резидент/нерезидент и опечатками. Вручную свести это в единую базу невозможно.

📥 Вход

ZIP-архив: PDF (текст и скан), DOCX (с правками), XLSX, XLS. Один файл = прайс одной клиники на дату.

⚙️ Что делаем

Извлекаем → нормализуем к справочнику → проверяем → версионируем историю цен.

📤 Выход

Единая БД + REST API/Swagger: поиск услуги, список клиник с ценами, очередь ручной разметки.

2Как данные проходят через систему (пайплайн)

Каждый файл проходит конвейер. Один «плохой» документ не ломает остальные — ошибки пишутся в лог документа.

3Извлечение по форматам (самое сложное — 30% оценки)

📄 PDF — адаптивно

Все 6 примеров — сканы с текстовым слоем разного качества. Логика: пробуем текстовый слой (таблицы → позиции слов → построчный парсер); если страница нечитаема — рендерим и распознаём Tesseract (rus+kaz+eng). Решение принимается по каждой странице, с лимитом OCR на документ.

📝 DOCX — с правками

Читаем XML напрямую: принимаем tracked changes (вставки оставляем, удаления отбрасываем). Достаём таблицы, секции «Раздел N» → категории.

📊 XLSX / XLS

Обходим все листы. Ищем строку-шапку (она спрятана под 8–17 строками «шапки-бланка» с «Утверждаю»). Объединённые ячейки, многострочные заголовки, строки-категории.

🧩 Плагин-архитектура

Один интерфейс BaseExtractor + реестр. Новый формат = один новый файл, ядро не меняется (требование масштабируемости ТЗ).

💡 Хитрые случаи из реальных данных: цены вида «14 400» (пробел-разделитель), 4 колонки цен вместо 2, шапка «для граждан РК» без слова «цена» (определяем колонку по числовому содержимому), бесшапочные нумерованные списки с ценами в конце строки.

4Модель данных

Ключевое отличие от ТЗ: цены вынесены в отдельную сущность PriceTier — потому что в реальных файлах до 4 тарифов, а не фиксированные «резидент/нерезидент».

Partner — клиника

  • code, display_name
  • legal_name (из шапки)
  • city, bin, контакты
  • is_active

PriceDocument — файл

  • partner_id, file_name
  • file_format, file_hash
  • status, parse_log
  • effective_date

PriceItem — позиция

  • raw_name, raw_code
  • service_id (нормализ.)
  • match_status, is_verified
  • is_active, superseded_by

PriceTier — тариф

  • tier_type
  • amount_kzt
  • amount_original, currency
  • label_raw (исходн.)
Partner ──< PriceDocument ──< PriceItem ──< PriceTier   •   PriceItem ──> Service ──< ServiceSynonym   •   MatchDecision (аудит решений)

5Нормализация к справочнику (25% оценки)

Загружаем справочник организаторов (Справочник услуг.xlsx, ~1230 услуг). Для каждой «сырой» услуги ищем лучшую запись справочника.

1️⃣ Нечёткий поиск

RapidFuzz по названиям и синонимам (нормализация текста: «консультация/осмотр» → «приём», аббревиатуры).

2️⃣ Смысловой поиск

Эмбеддинги (multilingual) по всему справочнику — ловит совпадения по смыслу, когда слова разные.

3️⃣ Решение по порогу

≥0.85 → авто; 0.60–0.85 → очередь ревью с подсказками; ниже → unmatched. Ручной матч учится в синонимы.

⚖️ Честно про метрику: ~37% авто при высокой точности. Снижать порог ради «70%» нельзя — ниже 0.85 совпадения в основном неверные (это испортит базу цен). ~92% позиций получают ранжированного кандидата → оператор подтверждает, а не ищет вручную. Рост авто-метрики = доменная модель эмбеддингов / больше синонимов, а не «подкрутка порога».

6Валидация и верификация (20% оценки)

Автопроверки (8 правил)

Цена > 0 · Нерезидент ≥ резидент · Имя не пустое · Дата не в будущем · Дубликат (клиника+услуга+дата) · Изменение цены >50% (аномалия) · Конвертация валют → KZT · Низкая уверенность OCR.

Ручная верификация

Очередь /unmatched с ранжированными подсказками → оператор подтверждает/исправляет/создаёт услугу через /match. Версионирование: старая цена не удаляется, а архивируется.

10Что дальше (приоритеты)

① Фронтенд-админка

Поиск → клиники с ценами, страница партнёра, загрузка/статусы, очередь верификации, дашборд. Закрывает раздел 4.6 (+10% оценки).

② Качество нормализации

Доменная модель эмбеддингов / расширение синонимов → выше авто-метрика без потери точности.

③ Метаданные + OCR

Вытаскивать город/контакты из шапок; deskew/denoise перед OCR для скан-PDF.
