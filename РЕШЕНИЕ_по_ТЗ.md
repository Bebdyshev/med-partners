## 2. Качество извлечения данных (критерий 30% · ТЗ 4.1–4.2)

### 2.1 Загрузка и первичная обработка (ТЗ 4.1)
- Приём **ZIP** или одиночного файла через UI и CLI; ZIP распаковывается, каждый файл — отдельный документ.
- **Автоопределение типа** по расширению + magic-bytes (`registry.detect_format`).
- **Очередь со статусом**: `queued → processing → done/needs_review/error`; обработка в фоне (Celery / daemon-поток), переживает перезагрузку вкладки.
- **Сохранность оригиналов**: файл копируется в `storage/raw_files/<uuid>/` и **никогда не удаляется** — повторная обработка возможна всегда. Дедуп по **SHA-256**.

### 2.2 Извлечение по формату (ТЗ 4.2) — плагинная архитектура
Каждый формат — отдельный `BaseExtractor`; новый формат = новый класс + строка в
`registry.py`, **ядро не меняется** (нефункц. требование «масштабируемость»).

| Формат | Как извлекаем |
|---|---|
| **PDF текстовый** | pdfplumber: ruled-таблицы → реконструкция таблицы по координатам слов (`words_to_table`) → headerless-парсер нумерованных списков. Решение «текст vs скан» — **по каждой странице** (порог символов + доля кириллицы). |
| **PDF скан** | страница растрируется (PyMuPDF) → **vision-LLM** возвращает СТРУКТУРИРОВАННЫЕ строки (имя / код / биоматериал / цены раздельно) — то, что теряет текстовый слой. **Tesseract** (`rus+kaz+eng`) как fallback; OCR-строки получают пониженную уверенность → смещение в очередь ревью. |
| **DOCX** | python-docx + чтение `word/document.xml`; **tracked changes** — принимается финальная версия (`<w:ins>` учитывается, `<w:del>` игнорируется). Извлечение всех таблиц. |
| **XLSX / XLS** | openpyxl / xlrd: обход **всех листов**, авто-поиск строки заголовков (не обязательно первой), извлечение колонок названия и цен. |

Каждая извлечённая строка несёт **`source_ref`** (`page=3;vision`, `sheet=Страховой;row=21`,
`table=2;row=5`) — для показа фрагмента-источника в ревью (ТЗ 4.4) и подсветки строки.

### 2.3 Разбор тарифов и валют
- Тип цены определяется по заголовку колонки (`tier_mapper`): `resident_kzt` (граждане РК),
  `near_abroad` (СНГ), `far_abroad`, `nonresident_generic`, `base_no_vat`.
- Сумма приводится к **₸** (`currency.to_kzt`) с сохранением оригинала и валюты
  (требование ТЗ 4.4 «Валюта не KZT → конвертировать по курсу, сохранить оригинал»).

## 3. Нормализация и сопоставление (критерий 25% · ТЗ 4.3)

`Matcher` (`app/normalization/engine.py`). Сигналы по убыванию точности:

1. **Code-first** — точное совпадение тарифного кода (`raw_code` или код, спрятанный в
   названии; учёт кириллических гомоглифов). → `auto`, score 1.0. Самый надёжный сигнал.
2. **Семантика** (если кода нет):
   - LLM-чистка названия перед эмбеддингом (`Узи орг.бр.пол+почки` → `УЗИ органов брюшной полости и почек`); справочник чистится так же.
   - эмбеддинг запроса → **косинус** ко всему справочнику; blended `max(cosine, fuzzy)` + **category boost**.
   - **LLM-судья** (`gpt-4o-mini`) проверяет top-k и даёт калиброванную уверенность — precision-стадия.
3. **Fuzzy fallback** (RapidFuzz) — если эмбеддинги/LLM недоступны (через circuit breaker, мгновенно).

**Пороги конфигурируемы** (ТЗ: «cosine > 0.85 → авто, иначе ревью»): `match_auto_threshold`,
`match_review_floor`, `llm_auto_threshold` в `.env`. Решение: `≥ auto` → **auto**;
`≥ floor` → **review** с ранжированными кандидатами; иначе → **unmatched**.

- **Очередь `unmatched`** — отдельная (`GET /unmatched`), для ручной разметки.
- **Ручное сопоставление** (`POST /match`) — оператор выбирает кандидата **или создаёт
  новую услугу** в справочнике; решение пишется в `MatchDecision` (аудит).
- **Цель ТЗ ≥70% автосопоставления** — превышена: **79.7%** на реальном архиве (auto 14 668
  из 18 415; review 5.0%, unmatched 15.3%). Метрики — в [`REPORT.md`](REPORT.md).
- **Точность auto** (precision-first): code-first **99.8%** (5 212 позиций с проверяемым кодом),
  семантическая стадия **79–81%** (EVAL.md), взвешенно ≈ **87%**. Auto-rate = покрытие,
  точность меряется отдельно по истине из тарифного кода.

## 4. Валидация и верификация ⭐ (критерий 20% · ТЗ 4.4)

### 4.1 Автоматические проверки — все из таблицы ТЗ 4.4
Правила **чистые, без ORM** (`app/validation/rules.py`) → юнит-тестируемы; конвейер
адаптирует `PriceItem`+тарифы в нейтральный `ValItem`.

| Правило (код) | Уровень | Пункт ТЗ 4.4 |
|---|---|---|
| `price_not_positive` | error | «цена > 0 и является числом» → `needs_review` |
| `nonresident_lt_resident` | warning | «цена нерезидента ≥ цены резидента» → флаг |
| `empty_name` | error | «название услуги не пустое» → пропуск/лог |
| `future_date` | warning | «дата прайса не в будущем» → предупреждение |
| `duplicate` | warning | «дублирование (клиника+услуга+дата)» → дедуп: новая сохраняется, старая архивируется |
| `price_anomaly` | warning | «изменение цены > 50%» → флаг аномалии, ручное подтверждение |
| `low_confidence` | warning | поддержка «документ без распознаваемых данных» → `error`; низкая уверенность OCR → ревью |

«Валюта не KZT» — закрыта на этапе разбора (конвертация в ₸ + сохранение оригинала, §2.3).
«Документ не содержит данных» → статус `error` + лог (`parse_log`).

**Связь с историей:** в `ValItem` пробрасывается `prev_resident_price` — цена резидента
**предыдущей активной версии** той же услуги у той же клиники. Аномалия считается
относительно реальной истории:

```
delta_pct = |new_resident − prev_resident| / prev_resident × 100
if delta_pct > PRICE_CHANGE_ANOMALY_PCT (по умолчанию 50%):  → price_anomaly
```

Любой флаг → позиция помечается, документ уходит в `needs_review`; warnings хранятся в
`PriceItem.warnings` (JSONB) и агрегируются в `PriceDocument.warnings` → видны в дашборде
и консоли ревью.

### 4.2 Версионирование — archive-on-change (ТЗ: «история цен хранится бессрочно»)
`app/services/versioning.py::supersede_previous`. Принцип: **ничего не удаляется**.

- **«Та же услуга»**: по `service_id` (если сопоставлена) или по нормализованному
  `raw_name` (если ещё `unmatched`) — у той же клиники.
- **Правило архивации** (защита от регресса — старый прайс не перезатрёт свежий):

```python
if new.effective_date is None or old.effective_date is None
   or old.effective_date <= new.effective_date:
       old.is_active = False
       old.superseded_by_id = new_item.id   # связь «кем заменена»
```

- **`is_active`** = актуальная версия (все витрины показывают её); цепочка
  `superseded_by_id` восстанавливает полную хронологию → мини-график «динамика цены» и
  блок «Цены по годам» на странице услуги; актуальные прайсы — сверху.
- Версионирование и валидация работают в паре: на каждой позиции сначала берётся
  `prev_resident_price`, выполняется валидация, затем `supersede_previous` делает новую
  версию активной, а прошлую — архивной.

### 4.3 Ручная верификация (ТЗ 4.4)
Консоль ревью one-at-a-time: **слева — исходный документ/фрагмент** (PDF-страница или
вырезка из Excel/Word вокруг строки по `source_ref`), **справа — извлечённые данные,
кандидаты справочника и ИИ-вердикт**. Оператор подтверждает / отклоняет / корректирует
(`POST /match`), есть массовое принятие по порогу (`/review/bulk-accept`).

## 5. API и поиск (критерий 15% · ТЗ 4.5)

**OpenAPI/Swagger** — автоматически (FastAPI, `/docs`). Все эндпоинты ТЗ 4.5 реализованы:

| Метод | Endpoint (ТЗ) | Статус |
|---|---|---|
| GET | `/services` (фильтр по категории) | ✅ |
| GET | `/services/{id}/partners` | ✅ |
| GET | `/partners` (фильтр по городу/статусу) | ✅ |
| GET | `/partners/{id}/services` | ✅ |
| GET | `/search?q=` | ✅ гибридный |
| GET | `/unmatched` | ✅ + кандидаты |
| POST | `/match` | ✅ выбор/создание услуги |

**Поиск** — трёхслойный гибрид (`search.py`): PostgreSQL **FTS** с двунаправленным
расширением аббревиатур (`ОАК ↔ общий анализ крови`) → **семантика** (эмбеддинги) при
малом числе результатов → **триграммы** (`pg_trgm`) на опечатки.

## TL;DR алгоритма

```
ZIP/файл → дедуп(SHA-256) → автоопределение формата
   → извлечение (плагин по формату: текст/таблицы/vision-OCR/tracked changes)
   → разбор позиций + тарифы + конвертация в ₸
   → нормализация (код → эмбеддинг → LLM-судья → fuzzy fallback; пороги конфиг.)
   → валидация (7 правил, аномалии относительно истории)
   → версионирование archive-on-change (актуальная версия + бессрочная история)
   → витрины: поиск (FTS+семантика+триграммы), дашборд, страницы услуг/клиник, ревью
```
Каждый критерий ТЗ закрыт; особое внимание — валидации и версионированию (§4).
